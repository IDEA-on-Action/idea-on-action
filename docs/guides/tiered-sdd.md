# Tiered SDD (Spec-Driven Development) 가이드

> **효율성과 안정성의 균형을 위한 개발 프로세스**
>
> 모든 변경사항에 무거운 SDD 프로세스를 적용하는 것은 비효율적입니다. 변경의 규모, 복잡도, 리스크에 따라 3단계(Tier)로 나누어 프로세스를 차등 적용합니다.

---

## 🎯 개요

**Tiered SDD**는 "문서화를 위한 문서화"를 지양하고, **"생각하고 코딩한다(Think before you code)"**는 본질을 지키면서 개발 속도를 최적화하기 위한 방법론입니다.

### 핵심 원칙
1. **리스크 기반 접근**: 리스크가 클수록 문서를 상세히 작성합니다.
2. **유연성**: 상황에 따라 적절한 Tier를 선택합니다.
3. **추적 가능성**: 어떤 Tier를 선택하든 "왜"와 "무엇"을 변경했는지 추적 가능해야 합니다.

---

## 📊 Tier 정의 및 워크플로우

### 🔴 Tier 1: Critical & Complex (Full SDD)

시스템의 핵심 구조를 변경하거나 높은 리스크를 수반하는 작업입니다. 가장 엄격한 프로세스를 따릅니다.

- **대상**:
  - 데이터베이스 스키마 변경 (테이블 생성/수정)
  - 핵심 비즈니스 로직 변경 (결제, 인증, 권한)
  - 대규모 신규 기능 개발 (새로운 페이지 그룹, 복잡한 워크플로우)
  - 아키텍처 변경 (라이브러리 교체, 폴더 구조 개편)
- **프로세스**:
  1. **Specify**: `spec/` 폴더에 상세 요구사항 및 기술 명세 작성
  2. **Plan**: `plan/` 폴더에 구현 계획 및 아키텍처 설계 작성
  3. **Tasks**: `tasks/` 폴더에 세부 작업 목록(To-Do) 작성
  4. **Implement**: 코드 구현 및 테스트
  5. **Verify**: 엄격한 테스트 및 검증
- **산출물**: Spec 문서, Plan 문서, Task 목록, 마이그레이션 파일, 테스트 코드

### 🟡 Tier 2: Enhancement & Standard (Lean SDD)

기존 기능을 개선하거나 중간 규모의 작업을 수행합니다. Spec 단계를 간소화하거나 생략합니다.

- **대상**:
  - 기존 기능의 확장 및 개선
  - UI/UX 컴포넌트 추가 및 수정
  - 비즈니스 로직의 마이너한 변경
  - 리팩토링 (기능 변경 없음)
- **프로세스**:
  1. **Plan & Tasks**: `plan/` 폴더에 간단한 설계와 작업 목록을 통합 작성 (또는 `tasks/`만 작성)
  2. **Implement**: 코드 구현
  3. **Verify**: 기능 테스트
- **산출물**: Plan/Task 문서 (통합 가능), 구현 코드

### 🟢 Tier 3: Trivial & Hotfix (Code-First)

단순 변경이나 긴급한 수정입니다. 별도의 문서 파일 없이 커밋 메시지와 PR 본문으로 대체합니다.

- **대상**:
  - 단순 버그 수정 (Hotfix)
  - 오타 수정, 문구 변경
  - 스타일(CSS) 단순 수정
  - 주석 추가, 문서 업데이트
  - 설정 파일의 사소한 변경
- **프로세스**:
  1. **Implement**: 즉시 코드 수정
  2. **Commit**: 커밋 메시지에 변경 이유와 내용 명시 (Conventional Commits 준수)
- **산출물**: 커밋 로그

## 🧭 Tier 결정 가이드 (Decision Matrix)

| 판단 기준 | Tier 1 (Critical) | Tier 2 (Standard) | Tier 3 (Trivial) |
| :--- | :--- | :--- | :--- |
| **DB 변경** | 있음 (스키마/데이터) | 없음 | 없음 |
| **영향 범위** | 시스템 전반 / 여러 모듈 | 단일 모듈 / 페이지 | 단일 파일 / 함수 |
| **리스크** | 데이터 손실 / 보안 / 결제 | 기능 오동작 / UX 저하 | 경미함 |
| **예상 시간** | 1일 이상 | 2시간 ~ 1일 | 2시간 미만 |
| **문서화** | `spec/` + `plan/` + `tasks/` | `plan/` (or `tasks/`) | Commit Message |

> **💡 Tip**: Tier가 애매할 때는 **상위 Tier**를 선택하는 것이 안전합니다.

## 📝 적용 예시

### Case 1: "결제 시스템에 정기 구독 기능을 추가하고 싶어"
- **판단**: DB 스키마 변경 필요, 결제 로직(핵심) 건드림, 리스크 높음.
- **결정**: **Tier 1**
- **Action**: `spec/subscription-system.md` 작성부터 시작.

### Case 2: "프로필 페이지 디자인을 좀 더 모던하게 바꾸자"
- **판단**: DB 변경 없음, UI 위주 작업, 영향 범위는 프로필 페이지 한정.
- **결정**: **Tier 2**
- **Action**: `plan/profile-redesign.md`에 변경안과 Task 정리 후 구현.

### Case 3: "로그인 버튼 색상이 브랜드 가이드랑 달라. 수정해줘."
- **판단**: 단순 CSS 수정, 리스크 낮음.
- **결정**: **Tier 3**
- **Action**: 즉시 수정 후 `fix(ui): correct login button color to match brand guide` 커밋.
